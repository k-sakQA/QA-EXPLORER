# QA Explorer Report Skill

QA Explorer (https://github.com/k-sakQA/QA-EXPLORER) のテスト結果を、シンプルで印刷もしやすい1枚のHTMLレポートに集約するためのskillです。

## このskillが対象とするリポジトリ構造

QA Explorer は以下の構造で結果を蓄積しています。このskillは、これらすべてを読み取って1枚のHTMLにまとめます。

```
qa-knowledge/
  targets/<target-slug>/
    findings.md              # Finding/Hypothesis/Probe の3層蓄積
    derived-viewpoints.md    # 派生観点
reports/
  <target-slug>/
    coverage.md              # 観点消化状況
    session-log.md           # セッション経緯 (Iteration 単位の記録)
    manual-tests.md          # 手動テスト候補 (存在する場合のみ)
    bugs/
      YYYY-MM-DD-viewpoint-NN-<slug>.md
tests/generated/
  *.spec.ts                  # Playwright spec
```

`<target-slug>` は対象URLのホスト名から生成された短い識別子（例: `hotel-example-site-takeyaqa-dev`)。複数の対象が並ぶこともある。

## ワークフロー

### Step 1: 対象を特定する

`reports/` 配下にあるすべての `<target-slug>` ディレクトリを列挙する。`_template/` ディレクトリは除外。

```bash
ls reports/ | grep -v '^_template$'
```

各 target-slug について、以下のファイルを読みに行く。

### Step 2: 各対象から情報を抽出する

target-slug ごとに、以下を読み取る。**ファイルが無い場合はスキップして空セクションとして扱う**(エラーにしない)。

| ファイル | 抽出する情報 |
|---|---|
| `reports/<slug>/bugs/*.md` | バグタイトル(H1)、Severity、Finding ID、観点、サマリー(冒頭2-3文) |
| `qa-knowledge/targets/<slug>/findings.md` | Severityが書かれていない bugs を補完するための Source/Viewpoint |
| `reports/<slug>/manual-tests.md` | 手動テスト候補(MT-ID, 優先度, タイトル, ファイル内アンカー)。ファイルが無ければ「起票なし」と表示 |
| `reports/<slug>/coverage.md` | 観点表(No, 観点, 優先度, 状態, 備考) |
| `reports/<slug>/session-log.md` | Iteration ごとのテスト記録 |
| `tests/generated/*.spec.ts` | spec のコメント(/** ~ */)から「狙うバグ」「手順」を補完 |

### Step 3: 「実行したテストの内容」をまとめる

これは2ソースを組み合わせる:

- **session-log.md の各 Iteration 記述** から「観点」「結果」「新規Finding/Hypothesis」を取る
- **対応する spec.ts のコメント** から「テスト意図」「手順」「期待値」を補完する

session-log の Iteration N と spec ファイルは、観点番号 (`viewpoint-NN-*.spec.ts` や `probe-YYYYMMDD-NN-*.spec.ts`) で対応付ける。1つの Iteration に1ブロック、`<details>` で折りたためるカードにまとめる。

各テストカードに含めるフィールド:
- **テスト意図** (spec のコメントから / 無ければ session-log の "狙い" から要約)
- **手順** (spec の test ブロックから抽出した操作を箇条書き / 細かすぎる場合は3-5ステップに要約)
- **期待値** (spec の expect 文や「期待:」コメントから)
- **結果** (session-log の Iteration 結果。FAIL / PASS / REJECTED で色分け)

### Step 4: HTMLを生成する

`references/template.html` をベースに、抽出した情報を埋め込む。テンプレートの構造は以下:

1. **ヘッダー** — タイトル + 生成日
2. **全体サマリー** — 4つのカード(対象数 / 検出バグ総数 / 手動テスト総数 / 平均カバレッジ)
3. **対象セクション** (対象ごとに繰り返し)
   - 検出バグ一覧
   - 手動テスト候補一覧
   - 観点カバレッジ表
   - 実行したテストの内容(`<details>`)
4. **フッター** — `Generated by QA Explorer Report Skill (Kaz Sakata)`

#### Severity と優先度の色分け対応

Severity と優先度はテンプレート内のCSSクラスに対応している:

- バグ Severity: `critical` / `high` / `medium` / `low` (バグレポートの「深刻度」「Severity」欄から取得。書かれていなければ `medium` をデフォルトに)
- 手動テスト 優先度: `high` / `medium` / `low`
- カバレッジ状態: `done` (消化済み/再消化済み/手動テスト完了) / `recheck` (再訪推奨/手動テスト待ち) / `todo` (未着手/実施中) / `na` (N/A)

#### 観点カバレッジ表に含める範囲

23観点すべてではなく、**「未着手以外」の観点 + 主要なN/A** を表示する。23行は冗長すぎる。ただし「未着手」の観点が多く残っている場合は、表の下に「他に N観点 が未着手」と注記する。

### Step 5: ファイルを保存して提示する

ファイル名は `qa-explorer-report-YYYYMMDD.html` 形式で、プロジェクトルートに保存する。

## 出力先

- 最終出力: `qa-explorer-report-YYYYMMDD.html`（プロジェクトルート）

## やらないこと

- **バグの修正提案や独自の品質評価をレポートに混ぜない**。このskillは「集約」が役割であり、reports/ に書かれていないことを推測で補わない。
- **観点リスト(viewpoints.md)の書き換えや派生観点の昇格判断はしない**。
- **テスト結果の評価コメント(良い/悪い)を勝手に追加しない**。事実(FAIL/PASS/REJECTED)だけを淡々と記載する。
- **ファイルが見つからないときに架空のデータで埋めない**。空セクションは「起票なし」と明示する。

## 参考ファイル

- `references/template.html` — HTMLレポートのテンプレート(完全な参照実装)。配色・レイアウト・印刷用CSSを含む。新しいレポートを作るときは、このファイルをコピーして中身のデータ部分だけ差し替える。
- `references/extraction-guide.md` — 各Markdownファイルから情報を抽出する具体的な方法(セクション見出しの目印、正規表現の例、エッジケースの扱い)。
