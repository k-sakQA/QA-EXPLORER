# QA Explorer

> **観点リスト**と**ナレッジ蓄積ループ**を使って、Webアプリケーションを自律的に探索テストするQAエージェント。
> VS Code + GitHub Copilot 上で動作します。

## このツールが目指すもの

従来の自動テストは「事前に書いた仕様を機械的に検証する」ものでした。
QA Explorer は逆で、**「何が壊れうるかをエージェント自身に考えさせながらテストする」** ための道具です。

具体的には、以下の3つを組み合わせて動きます:

1. **観点リスト** — QAの基本形を網羅した地図
2. **ナレッジ蓄積ループ (Finding → Hypothesis → Probe)** — 気づきを次のテストに活かす仕組み
3. **AI / 人間協業** — 自動化できないものは `manual-tests.md` に手順書を残して人間に委譲

結果として、1セッションで複数のバグを検出しつつ、**「このサイトは状態管理が弱い」「このタイプのサイトではこう壊れがち」** といったパターン知識が自動で蓄積されていきます。

---

## 動作の例

`/explore url=https://hotel-example-site.takeyaqa.dev/ja/plans.html intent=予約フォームを重点的に` を実行すると、エージェントは:

1. 観点を P0 から順に消化し始める
2. 観点10(単機能)で「ブラウザバックでメールが消える」バグを検出 → `F-20260422-01` として記録
3. 観点23(禁則)で「宿泊数0で予約可能」「人数0で予約可能」を検出 → `F-20260422-04,05`、仮説 `H-20260422-02` (min属性未設定) を立てて確認
4. 複数の Finding から「このサイトは状態管理が弱い」という仮説 `H-20260422-01` を立て、検証用 Probe を計画
5. Probe を実行して仮説を Confirmed にし、関連バグ `F-20260422-07` (日付・宿泊数もバックで消える) を追加発見
6. P3 観点(同時実行・別デバイス等)は自動化困難と判定し、`manual-tests.md` に手動テスト手順を起票
7. セッション終了時、`site-profiles.md` の「予約系サイト」プロファイルに発見を還元

次セッション開始時は `findings.md` を全件読み直すので、前回の仮説が未検証なら優先的に検証に入ります。

## 構成

```
qa-explorer/
├─ .github/
│  ├─ copilot-instructions.md           # Copilot の基本原則(常時適用)
│  └─ agents/
│     └─ qa-explorer.agent.md           # ルンバ型エージェント本体
├─ .prompts/
│  └─ explore.prompt.md                 # /explore で起動するプロンプト
├─ qa-knowledge/
│  ├─ viewpoints.md                     # 観点リスト(基本形・不変)
│  ├─ site-profiles.md                  # サイトタイプ別の着目ポイント(育っていく)
│  └─ targets/
│     ├─ _template/                     # 新規対象用のひな形
│     └─ <target-slug>/                 # 対象ごとのナレッジ
│        ├─ findings.md                 # Finding / Hypothesis / Probe の3層蓄積
│        └─ derived-viewpoints.md       # 対象固有の派生観点
├─ reports/
│  ├─ _template/                        # 新規対象用のひな形
│  └─ <target-slug>/                    # 対象ごとのレポート
│     ├─ coverage.md                    # 観点消化状況
│     ├─ session-log.md                 # セッションの経緯
│     ├─ manual-tests.md                # 人間が実行すべきテスト(優先度付き)
│     └─ bugs/                          # GitHub Issue 形式のバグレポート
└─ tests/generated/                     # エージェントが生成する Playwright spec
```

`<target-slug>` は対象URLのホスト名から自動生成されます。
例: `hotel-example-site.takeyaqa.dev` → `hotel-example-site-takeyaqa-dev`

## セットアップ

### 前提

- VS Code
- GitHub Copilot 拡張機能 (Agent モード利用可)
- Node.js 18 以上

### インストール

```bash
git clone https://github.com/k-sakQA/QA-EXPLORER.git
cd QA-EXPLORER
npm install
npx playwright install
```

VS Code でリポジトリを開き、Copilot Chat を Agent モードに切り替えます。

### (推奨) Copilot のエージェント最大リクエスト数を増やす

`.vscode/settings.json` で `chat.agent.maxRequests` が 100 に設定されています。
1セッションで数十回のツール呼び出しが必要になるため、この値は変更せずに使うことをお勧めします。

## 使い方

### 基本コマンド

VS Code の Copilot Chat (Agent モード) で:

```
/explore url=<URL> intent=<テスト意図>
```

入力例:

```
/explore url=https://hotel-example-site.takeyaqa.dev/ja/plans.html intent=予約フォームのバリデーションを重点的に
```

### 止め方

チャットで「Stop」「止めて」「中断」のいずれかを入力すると、
現在のイテレーションを完了させてから、セッション終了処理を実行して止まります。

### セッションを跨いで続きから再開

同じ URL で再度 `/explore` を実行すると、エージェントは既存の `findings.md` を
**全件読み直してから** 探索を始めます。前回立てた仮説のうち未検証のものがあれば、
そこから優先的に検証に入ります。

## 3層のデータモデル

### Finding (F-YYYYMMDD-NN)

テスト中に観察した事実。バグ・気づき・仮説検証の結果を記録します。
Source は `Bug` / `Observation` / `Hypothesis-Confirmed` / `Hypothesis-Rejected` のいずれか。

### Hypothesis (H-YYYYMMDD-NN)

複数の Finding から立てた、対象の弱点や実装上の癖に関する仮説。
**2件以上の Finding が同じ匂い**(原因領域、影響するユーザー操作、UI要素の種類)を放っていたら自動的に立てます。
Status は `Open` / `Probing` / `Confirmed` / `Rejected`。

### Probe (P-YYYYMMDD-NN)

仮説を検証するための具体的なテスト計画。
Planned 状態で登録され、実行後に Done になって Hypothesis の Status を更新します。

---

## 観点の状態管理

`coverage.md` の状態欄は単なる「済 / 未」ではなく、探索の状況を細かく表します:

| 状態 | 意味 |
|---|---|
| 未着手 | まだ実施していない |
| 実施中 | 現在進行中 |
| 消化済み | 一度実施した(基本パスのみ) |
| 再訪推奨 | 消化済みだが Hypothesis により再テスト推奨 |
| 再消化済み | 再訪推奨に応じて再実施完了 |
| 手動テスト待ち | エージェントが manual-tests.md に手順書を起票済み |
| 手動テスト完了 | 人間が実行し、結果を記入済み |
| N/A | この対象では対象外 |

「再訪推奨」は、1度消化した観点でも Hypothesis が立った時点で再度スポットライトが当たる仕組みです。

## AI と人間の役割分担

このツールは **全自動** ではありません。自動化できないものは正直に人間に渡します。

| 領域 | 担当 |
|---|---|
| HTML/JS で再現できる振る舞い、バリデーション、遷移、DOM状態 | AI (Playwright) |
| 同時実行 / レースコンディション | 人間 (manual-tests.md) |
| 実機モバイルUX (タッチ、ソフトキーボード、日付ピッカー) | 人間 |
| スクリーンリーダーでの主観評価 | 人間 |
| 長時間放置・大量データ | 人間 |
| 決済・本物のメール送信 | 人間 |

エージェントが「自動化困難」と判定したものは、`manual-tests.md` に優先度 (High/Medium/Low) 付きの手順書として起票されます。
人間が実機で検証し、Fail だった場合は通常のバグと同じフロー(`bugs/` への起票、`findings.md` への登録)で扱います。

## サイトプロファイル

`qa-knowledge/site-profiles.md` は **対象タイプ別の着目ポイント** のカタログです。
**観点リストには含めません**。対象を見極める「気づきの種」として使います。

エージェントは起動時に対象URLとDOMからプロファイルを照合し、マッチすれば:

- あらかじめ決められた観点に加えて、そのプロファイルの「重点ポイント」を探索計画に織り込む
- 過去に同タイプの対象で立った Hypothesis を参考に、効率的に仮説を立てる

例えば「予約系サイト」プロファイルには、以下が記録されています:

- 条件付き必須UI × ブラウザ履歴操作
- 数値入力の min/max 属性と JS バリデーションの二重化
- 全角スペース等の空白文字を含む必須チェック

マッチしなくても問題ありません。プロファイルは加点要素です。

## バグレポート形式

`reports/<target-slug>/bugs/` に生成されるバグレポートは、**GitHub Issue にそのまま貼り付けられる形式** です。

```markdown
# <タイトル>

## Steps
## As is
## To be
## 再現箇所・URL
## 関連するIssue
## 対象テストケース
## 不具合が発生している環境 - 再現端末(OS・ブラウザ)
- Web
  - [ ] iOS:
  - [ ] Android:
- PWA
  - [ ] iOS:
  - [ ] Android:
## 備考
(Finding ID / Hypothesis / Severity / Date 等のメタ情報)
```

エージェントが Playwright (Chromium) で検出したバグは、環境欄が全項目 `未確認` で起票されます。
モバイル実機での再現確認が必要な場合は、後から人間がチェックを入れて端末名を記入します。

## 観点リストについて

`qa-knowledge/viewpoints.md` は **基本形** であり、**個別対象の都合で書き換えません**。
対象ごとの特殊な観点は `derived-viewpoints.md` に `DV-YYYYMMDD-NN` として記録します。

観点一覧:

| No | カテゴリ | 観点 | 優先度 |
|---:|---------|-----|:----:|
| 1 | 表示(UI) | レイアウト/文言 | P2 |
| 2 | 表示(UI) | エラー表示(正常系) | P0 |
| 3 | 入力 | 文字種 | P1 |
| 4 | 入力 | 文字数(正常値) | P2 |
| 5 | 入力 | 文字数(正常限界) | P2 |
| 6 | 入力 | 文字数(異常値) | P2 |
| 7 | 入力 | 数値(正常値) | P1 |
| 8 | 入力 | 数値(異常値) | P2 |
| 9 | 入力 | 未入力 | P2 |
| 10 | 動作確認 | 単機能 | P0 |
| 11 | 状態遷移 | イベントによる状態変化 | P0 |
| 12 | 状態遷移 | 経時変化 | P2 |
| 13 | 画面遷移 | 切替えとデータ保持 | P2 |
| 14 | 設定保持 | 初期値 | P2 |
| 15 | 設定保持 | 変更・反映 | P2 |
| 16 | 設定保持 | 設定保持 | N/A |
| 17 | 設定保持 | キャンセル | P2 |
| 18 | 機能組合せ | 複数因子の組合せ | P2 |
| 19 | 同時実行 | 登録と参照 | P3 |
| 20 | 同時実行 | 複数アカウント | P3 |
| 21 | 割込み | 別デバイス | P3 |
| 22 | 割込み | 登録と参照 | P3 |
| 23 | 排他処理 | 禁則 | P0 |

消化順序は `P0 → P1 → P2 → P3`。ただし Planned Probe や再訪推奨があれば、そちらが最優先です。

## このツールが「やらない」こと

- **バグの修正** — QAは原因を見つけて報告する役割。修正は開発者の仕事です。
- **本番DBへの書き込み・本物の決済・本物のメール送信** — テストデータは必ず架空のものを使います。
- **観点リストの自動書き換え** — 基本形(`viewpoints.md`)はユーザーが管理します。派生観点 / プロファイル項目が複数対象で有効だと分かった場合は、エージェントが昇格を **提案** しますが、勝手に昇格はしません。
- **自動化できないテストを無理に Playwright で書くこと** — 潔く `manual-tests.md` に回します。
- **実機確認していないバグの環境欄にチェックを入れること** — 自動検出したものは `未確認` と明記します。

---

## 安全面の前提

- 実在する人物の個人情報・本物のクレジットカード番号・本物の認証情報をテストデータに使いません
- ログイン情報やAPIキーはコードに直書きせず、必要なら環境変数経由で扱います
- 対象URLに `prod` / `production` が含まれる等、本番環境の可能性がある場合は、最初の1回だけユーザーに確認を取ります

---

## ツールの拡張について

このツールは **観点リスト + ナレッジ + プロファイル** の組み合わせで成り立っています。
テスト対象が増えるほど、プロファイルが育ち、同種のサイトに対して次回以降の効率が上がる設計です。

以下は任意で追加できます:

- `qa-knowledge/test-patterns.md` — 観点ごとの具体的なテストパターン集
- `qa-knowledge/bug-templates.md` — バグ起票のフォーマット(現状は `reports/_template/bugs/_bug-report-template.md` で代替)
